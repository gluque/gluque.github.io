1 What is Django?

1.1 It isn't neither a singer nor a free slave

  • Django is a framework to develop web applications in Python. Period.

2 Installation

2.1 MySQL

  • Install Maria DB (a MySQL fork with steroids). During installation, you have to assign a root's password.
  • From the command line, access to the MariaDB server: mysql -u root -p
CREATE DATABASE trydjango CHARACTER SET utf8;

2.2 Python 3.5

  • I'd prefer to use Anaconda's bundle. So, download it from here and follow the install instructions according to your OS.

2.3 Virtual Environment

  • Once Anaconda has been installed, create a new virtual environment.
  • This is not mandatory, but it will allow you to have an isolated Python's local installation.
  • Create environment: conda create -n djangodev django
  • Activate environment: source activate djangodev
  • Deactivate environment: introduce exit in your command line or close it

2.4 Additional Python's packages

  • MySQL (MariaDB) driver: conda install -n djangodev pymysl

3 First steps

3.1 New project

  • Here we are going to create a typical blog application. Nothing astonishing but it will serve to understand the big picture.
  • First, activate your environment
  • Second, create your project: django-admin startproject trydjango
  • In Django, a project is a set of applications. One of these will be our blog, but we can also have an admin control panel for example.
  • Remember: one project, many apps.

3.2 Database connection

  • Update your setting's file (trydjango/settings.py):
# after *import os* put this:
import pymysql
pymysql.install_as_MySQLdb()

# ... many lines of code here ...

# now update the default database connection info:
DATABASES = {
    'default': {
        'ENGINE': 'django.db.backends.mysql',
        'NAME': 'trydjango',
        'USER': 'root',
        'PASSWORD': 'your_password',
        'HOST': 'localhost',
        'PORT': '',
    }
}

3.3 First migration

  • In Django lingua, a migration means to update your database schema with the changes made in the source code.
  • By default, Django include some pre-built applications inside such as admin.
  • From this application, you can manage the users of your website and create nice CRUD interfaces.
  • In the command line, exec this: python manage.py migrate
  • Create a superuser: python manage.py createsuperuser
  • Now you can run your Django's web server (yeah, batteries included!): python manage.py runserver

3.4 Creating the posts app

  • In the command line: python manage.py startapp posts
  • Django's apps follow a Model-View-Controller pattern.
  • Open posts/models.py and define a model to host the posts user enters:
class Post(models.Model):
    title = models.CharField(max_length=120)
    content = models.TextField()
    updated = models.DateTimeField(auto_now=True, auto_now_add=False)
    timestamp = models.DateTimeField(auto_now=False, auto_now_add=True)

    def __str__(self):
        return self.title
  • You need to add the application to the project, so edit the list of INSTALLED_APPS in the trydjango/settings.py file:
INSTALLED_APPS = [
    'django.contrib.admin',
    'django.contrib.auth',
    'django.contrib.contenttypes',
    'django.contrib.sessions',
    'django.contrib.messages',
    'django.contrib.staticfiles',
    'posts'
]

3.5 Update the database

  • Each time a new model is added or changed, you have to:
  • create the migration files: python manage.py makemigrations
  • and update the database's scheme: python manage.py migrate

3.6 Model in control panel

  • The previous model has 4 fields, which were declared using 3 distinct model field references. Django has a lot of them, just check at: Model Fields
  • The admin application works like a control or admin panel for your website. For now you can add new users or groups, but you can also put them the models you define.
  • All the objects in the admin application inherited their functionality from the admin.ModelAdmin class and are linked to either a built-in or a handcrafted model. See at ModelAdmin Objects.
  • We have to declare a new ModelAdmin class and connect it to our posts model, so we can customize how the model is presented to the user.
  • Modify posts/admin.py to register your model in the admin panel:
from .models import Post

class PostsModelAdmin(admin.ModelAdmin):
    list_display = ["__str__", "updated", "timestamp"]
    list_display_links = ["updated"]
    list_filter = ["updated", "timestamp"]
    search_fields = ["title", "content"]
    list_editable = ["title"]
    class Meta:
        model = Post


admin.site.register(Post, PostsModelAdmin)

4 Views and navigation in Django apps

4.1 CRUD

  • C = Create = POST -> Create views (a new post)
  • R = Retrieve data = GET -> List views
  • U = Update = PUT/PATCH -> Update views (locating an existing post)
  • D = Delete = DELETE -> Delete views (with confirmation)
  • You are always in an REQUEST-RESPONSE cycle when surfing on the web.
  • What if we want to make our own pages to modify/view posts?, I mean, instead of have to use the admin application.

4.2 Introducing function based views

  • Let's create a simple view to list the posts we have in the database.
  • This is the simplest way to create views in Django. They're just functions.
  • Edit the posts/views.py file:
from django.shortcuts import render
from django.http import HttpResponse

# Create your views here.

def posts_home(render):
    return HttpResponse("<h1>Hello</h1")

4.3 URLs (I)

  • Once we have a simple view (or a complex one, later), we should create an URL pattern so we can send the request to the function and wait for its response.
  • Django URL patterns are a mapping of regular expressions to view functions or classes. Please see this link for more examples.
  • Add a new URL pattern to the trydjango/urls.py file:
from django.conf.urls import url
from django.contrib import admin

urlpatterns = [
    url(r'^admin/', admin.site.urls),
    url(r'^posts/$', "posts.views.posts_home")
    # url(r'^posts/$', "<app_name>.views.<function_name>")
]

4.4 URLs (II)

  • Nevertheless there is a better way to manage the applications's URL mappings to navigate through it.
  • Firstly, create fake view functions for each of the CRUD operations in the posts/views.py file:
from django.shortcuts import render
from django.http import HttpResponse

# Create your views here.

def post_list(request):
    return HttpResponse("<h1>List</h1")

def post_create(request):
    return HttpResponse("<h1>Create</h1")

def post_detail(request):
    return HttpResponse("<h1>Detail</h1")

def post_update(request):
    return HttpResponse("<h1>Update</h1")

def post_delete(request):
    return HttpResponse("<h1>Delete</h1")
  • Next, create a new file called posts/urls.py and define mappings to each view:
from django.conf.urls import url
import posts.views

urlpatterns = [
    url(r'^$', posts.views.post_list),
    url(r'^create/$', posts.views.post_create),
    url(r'^detail/$', posts.views.post_detail),
    url(r'^update/$', posts.views.post_update),
    url(r'^delete/$', posts.views.post_delete)
]
  • Finally, update the trydjango/urls.py file to include the already created file:
from django.conf.urls import url, include
from django.contrib import admin

urlpatterns = [
    url(r'^admin/', admin.site.urls),
    url(r'^posts/', include("posts.urls"))
    # url(r'^posts/$', "<app_name>.views.<function_name>")
]

5 Templates

5.1 Base directory

  • In your trydjango/settings.py there is a veriable called BASE_DIR.
  • BASE_DIR refers to the base directory of your project, i.e. where manage.py file resides.
  • Using the os Python's library, Django can know its relative path in spite of the subyacent operating system.

5.2 Setting templates

  • In your project directory create a new directory called templates
  • Edit your trydjango/settings.py file and set the location of your templates directory using BASE_DIR:
TEMPLATES = [
    {
        'BACKEND': 'django.template.backends.django.DjangoTemplates',
        'DIRS': [os.path.join(BASE_DIR, "templates")],
        'APP_DIRS': True,
        'OPTIONS': {
            'context_processors': [
                'django.template.context_processors.debug',
                'django.template.context_processors.request',
                'django.contrib.auth.context_processors.auth',
                'django.contrib.messages.context_processors.messages',
            ],
        },
    },
]

5.3 Using templates

  • A view function can use the render function to take the request, looks for a template and passing a Python's dictionary in there.
  • Templates are basically HTML files located at the templates folder.
  • In a view, we refer to a template only by its filename. The directory where the file is stored, is located automatically by Django thanks to the settings adjustments we have done.
  • Let's create a sample template index.html in the templates directory:
<!-- DOCTYPE html -->
<html>
    <body>
        <h1>{{ title }}</h1>
        <ul>{% for obj in object_list %}
            <li>{{ obj.title }}</li>
            {% endfor %}
        </ul>
    </body>
</html>
  • Now we can modify the post list view in the posts/views.py file:
from django.shortcuts import render
from django.http import HttpResponse
from posts.models import Post

# Create your views here.

def post_list(render):
    queryset = Posts.objects.all()
    context = {
        'title': 'Posts List',
        'object_list': queryset
    }
    return render(request, 'index.html', context)
